Hello guys and welcome! I've been recently looking at Android with their fancy ol' "Winlator" and wanted to build something like that **but for iOS**. Below, you'll find the breakdown of how the Claw Porting Engine works, the exact translation pipelines used, and what to expect going forward.
### 🛠️ What is Claw Porting Engine?
**Claw Porting Engine (CPE)** is an iOS/iPadOS porting engine designed to allow Windows games to run directly on your Apple devices.
 * **The Core:** It spins up a custom, GUI-less Linux subsystem.
 * **The Drivers:** It executes **Wine** and **Box64** inside this environment.
 * **The Layer:** A multi-process translation stack coordinates communication between the game binary and the host Linux container (**Wine** ──> **Box64** ──> **VirGL/Mesa** ──> **Metal API**) to make rendering possible.
### ⚖️ What is The Difference between CPE And GPTK?
| Feature | Apple GPTK | Claw Porting Engine (CPE) |
|---|---|---|
| **Approach** | Native API Translation Layer | Virtualized Linux Subsystem Container |
| **Emulation / JIT** | None (Compiles natively for Apple Silicon) | Emulated via Just-in-Time (**Box64 / FEX JIT**) |
| **Current Performance** | Highly integrated but can experience optimization lag | Heavy rendering overhead (Active development optimizing frame-by-frame processing) |
| **Target Design** | Intended for developers porting source projects | Built to play existing x86/x64 PC titles on iDevices |
While GPTK is built directly into the host OS framework, it remains a closed ecosystem and can be unoptimized for certain standard PC binaries. CPE bridges that gap by using high-performance JIT execution mechanics inside a specialized Linux stack. Though it isn't completely native, the goal is to optimize this container pipeline as much as humanly possible until it becomes the ultimate way to play PC games on your iOS hardware.

### 🔄 What is the Process of Translation Expected in the Release?
The definitive deployment architecture utilizes an explicit execution pipeline to map Windows system instructions over to the iOS surface area:
```
[ Windows Game (.exe) ] ──> [ Wine ] ──> [ Box64 JIT ] ──> [ VirGL / Mesa ] ──> [ Metal API ] ──> iOS Screen

```
Because this multi-layered translation stack must execute sequentially for every single frame generated by the game, it introduces significant rendering overhead and latency.
To solve this, we are currently developing a custom engine component called **Iron Interceptor**.
 * **The Goal:** Completely bypass the need for VirGL to pass generic OpenGL/Vulkan commands over a virtualized screen link.
 * **The Fix:** Iron Interceptor hooks straight into Windows graphics API calls (d3d9/d3d11/d3d12) directly inside Wine, parsing and converting them into native **Metal-friendly command buffers** before they ever leave the container.
 * **The Result:** Eliminates multiple intermediate layers, turning a heavy, multi-staged pipeline into a direct, high-performance rendering engine.
### ⚡ What Translation Process Will Be Used in the Pre-Release?
To maximize speed and performance right out of the gate in the pre-release, the stack shifts to a pre-built compatibility layout using a Steam-centric compositor pipeline:
```
[ Steam Gamepad UI ] ──> [ Proton / FEX JIT ] ──> [ Gamescope Compositor ] ──> [ VirGL / Mesa ] ──> [ Metal API ] ──> iOS Screen

```
This alternative pipeline runs Steam alongside its native **Proton** translation layer to translate the Windows API hooks straight down into Vulkan (which is supported out-of-the-box by Linux backend utilities). From there, **Mesa** or **VirGL** parses those instructions into structured commands that map clean into Apple's native Metal framework.
### 🚀 What to Expect in the Release?
 * **User Experience:** A clean, minimal, user-friendly mobile application interface.
 * **Compatibility:** Smooth compatibility across mid-tier PC titles (e.g., *Counter-Strike*).
 * **Limitations:** Massive AAA titles like *Red Dead Redemption 2*, *The Last Of Us Remastered*, or *GTA V* will not run reliably due to extreme hardware limits on modern mobile systems.
 * **Control Scheme:** A hardware **Keyboard and Mouse** is heavily recommended for the best possible interaction support.
